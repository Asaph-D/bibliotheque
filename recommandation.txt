API GraphQL Bibliothèque Technologie : Spring Boot 3 + Spring for GraphQL Objectif : Développer une API GraphQL complète pour une bibliothèque : livres, auteurs, emprunts. Illustrer la resolution du probleme d'over-fetching et d'under fetching de REST, les mutations et les subscriptions temps reel. Durée estimée : 2 heures 30 Contexte et objectifs pédagogiques GraphQL repond aux limitations de REST : l'over-fetching (recevoir trop de champs) et l'under fetching (faire plusieurs requêtes pour assembler les données). Dans ce TP, vous comparerez concrètement REST vs GraphQL sur le meme jeu de données, puis vous implémenterez Queries, Mutations et Subscriptions avec Spring for GraphQL. Concepts couverts : • Schéma Definition Language (SDL) GraphQL • Queries : lecture de données avec selection de champs • Mutations : creation, modification, suppression • Subscriptions : notifications temps reel (WebSocket) • DataLoader : eviter le probleme N+1 requêtes en base Schéma GraphQL (src/main/resources/graphql/schéma.graphqls) type Query { # Recherche de livres avec filtres optionnels livres(titre: String, genre: Genre, disponible: Boolean): [Livre!]! # Detail d'un livre avec ses relations livre(id: ID!): Livre # Liste des auteurs auteurs(nom: String): [Auteur!]! } type Mutation { # Emprunter un livre (retourne l'emprunt créé) emprunterLivre(livreId: ID!, adherentId: ID!): Emprunt! # Retourner un livre retournerLivre(empruntId: ID!): Emprunt! # Ajouter un nouveau livre (admin) ajouterLivre(input: LivreInput!): Livre! } type Subscription { # Notifier en temps reel quand un livre devient disponible livreDisponible(genre: Genre): Livre! } type Livre { id: ID! titre: String! isbn: String! genre: Genre! anneePublication: Int disponible: Boolean! auteur: Auteur! # relation resolvee separement (DataLoader) empruntsActifs: [Emprunt!]! } type Auteur { id: ID! nom: String! nationalite: String livres: [Livre!]! # relation inverse } type Emprunt { id: ID! dateEmprunt: String! dateRetourPrevue: String! dateRetourEffective: String livre: Livre! adherent: Adherent! } input LivreInput { titre: String! isbn: String! genre: Genre! auteurId: ID! anneePublication: Int } enum Genre { ROMAN SCIENCE_FICTION POLICIER BIOGRAPHIE HISTOIRE SCIENCE } Étapes de réalisation Étape 1 Configurer Spring for GraphQL <!-- pom.xml --> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-graphql</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-websocket</artifactId> <!-- necessaire pour les Subscriptions --> </dependency> # application.properties spring.graphql.path=/graphql spring.graphql.graphiql.enabled=true # interface web de test spring.graphql.graphiql.path=/graphiql spring.graphql.websocket.path=/graphql-ws # pour les Subscriptions Étape 2 Implémenter les resolvers GraphQL @Controller public class LivreController { @Autowired private LivreService livreService; @Autowired private EmpruntService empruntService; // === QUERIES === @QueryMapping public List<Livre> livres( @Argument String titre, @Argument Genre genre, @Argument Boolean disponible) { return livreService.rechercher(titre, genre, disponible); } @QueryMapping public Optional<Livre> livre(@Argument Long id) { return livreService.findById(id); } // === MUTATION === @MutationMapping public Emprunt emprunterLivre( @Argument Long livreId, @Argument Long adherentId) { return empruntService.emprunter(livreId, adherentId); } @MutationMapping public Emprunt retournerLivre(@Argument Long empruntId) { Emprunt retour = empruntService.retourner(empruntId); // Notifier les subscribers que le livre est disponible livreDisponibleSink.tryEmitNext(retour.getLivre()); return retour; } // === SUBSCRIPTION === private final Sinks.Many<Livre> livreDisponibleSink = Sinks.many().multicast().onBackpressureBuffer(); @SubscriptionMapping public Flux<Livre> livreDisponible(@Argument Genre genre) { return livreDisponibleSink.asFlux() .filter(l -> genre == null || l.getGenre() == genre); } } Étape 3 Implémenter le DataLoader (probleme N+1) Sans DataLoader, charger 100 livres avec leurs auteurs = 1 requête livres + 100 requêtes auteurs = probleme N+1. Avec DataLoader, c'est 1 + 1 requêtes. @Component public class AuteurDataLoader implements BatchLoaderWithContext<Long, Auteur> { @Autowired private AuteurRepository auteurRepo; @Override public CompletionStage<List<Auteur>> load( List<Long> auteurIds, BatchLoaderEnvironment env) { // UNE SEULE requête SQL pour tous les auteurs demandes // au lieu de N requêtes individuelles List<Auteur> auteurs = auteurRepo.findAllById(auteurIds); // Retourner dans le meme ordre que les IDs demandes Map<Long, Auteur> map = auteurs.stream() .collect(Collectors.toMap(Auteur::getId, a -> a)); return CompletableFuture.complètedFuture( auteurIds.stream().map(map::get).toList()); } } // Declarer le DataLoader dans un @Bean : @Bean public RuntimeWiringConfigurer runtimeWiringConfigurer( AuteurDataLoader auteurLoader) { return builder -> builder.type("Livre", b -> b.dataFetcher("auteur", env -> { DataLoader<Long, Auteur> loader = env.getDataLoader("auteurs"); Long auteurId = env.<Livre>getSource().getAuteurId(); return loader.load(auteurId); // batch automatique })); } Étape 4 Tester avec GraphiQL et comparer avec REST Ouvrez http://localhost:8080/graphiql et executez ces requêtes : # Requête 1 : seulement titre + disponibilite (evite l'over-fetching) query { livres(genre: ROMAN, disponible: true) { id titre disponible } } # Requête 2 : titre + auteur en une seule requête (evite l'under-fetching) query { livres { titre auteur { nom nationalite } empruntsActifs { dateRetourPrevue adherent { nom } } } } # Mutation : emprunter un livre mutation { emprunterLivre(livreId: "1", adherentId: "42") { id dateRetourPrevue livre { titre } } } # Subscription (depuis un onglet separé dans GraphiQL) : subscription { livreDisponible(genre: ROMAN) { titre auteur { nom } } } # Puis effectuez un retour de livre pour voir la notification en temps reel TIP Comparez la taille des réponses REST vs GraphQL : avec REST /api/livres, tous les champs sont retournes (~40 champs). Avec GraphQL en demandant seulement id + titre + disponible, la réponse est 10x plus petite. Mesurez la difference avec curl -w '%{size_download}'.
